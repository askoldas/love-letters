# Database Schema

Database: Supabase Postgres.

This document describes the intended MVP schema. Exact SQL can be adapted during implementation.

## Design principles

- Public visitors do not have accounts.
- Admins use Supabase Auth.
- Every submitted letter gets an archive number immediately.
- Letter status controls visibility.
- Email is optional and not public.
- No token-based private management in MVP.
- Likes are hearts, not 1-10 ratings.
- Liked letters act as saved/bookmarked letters for that visitor.

## `letters`

Main table for all submitted letters.

Suggested fields:

```sql
create table public.letters (
  id uuid primary key default gen_random_uuid(),

  archive_number bigint generated by default as identity unique not null,
  public_slug text generated always as ('letter-' || archive_number::text) stored,

  text text not null,
  author_name text,
  is_anonymous boolean not null default false,

  city text,
  country text,
  language text,

  email text,
  email_normalized text,

  status text not null default 'pending'
    check (status in ('pending', 'approved', 'rejected', 'hidden')),

  hearts_count integer not null default 0,
  shares_count integer not null default 0,
  is_featured boolean not null default false,

  admin_notes text,
  rejection_reason text,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  approved_at timestamptz,
  rejected_at timestamptz,
  hidden_at timestamptz
);
```

### Notes

`archive_number` is assigned immediately. Display it as:

```txt
№ 000127
```

Use formatting in application code, not in DB.

`email` must never be displayed publicly.

`email_normalized` can store lowercased/trimmed email for lookup. If more privacy is needed later, also store an email hash and avoid querying directly by raw email.

## `letter_likes`

Tracks hearts/likes and prevents repeated likes by the same visitor.

```sql
create table public.letter_likes (
  id uuid primary key default gen_random_uuid(),
  letter_id uuid not null references public.letters(id) on delete cascade,
  visitor_id text not null,
  created_at timestamptz not null default now(),

  unique (letter_id, visitor_id)
);
```

### Visitor ID

Because public users do not register, create anonymous visitor ID in cookie/localStorage.

Example:

```txt
ll_visitor_id=<random uuid>
```

Server still needs rate limiting because client-side IDs can be reset.

## `letter_shares`

Optional table to track share actions.

```sql
create table public.letter_shares (
  id uuid primary key default gen_random_uuid(),
  letter_id uuid not null references public.letters(id) on delete cascade,
  visitor_id text,
  platform text,
  created_at timestamptz not null default now()
);
```

`shares_count` can be incremented on `letters` when a share action is triggered.

MVP can start with only `shares_count` if detailed analytics are not needed.

## `admin_profiles`

Maps Supabase Auth users to admin roles.

```sql
create table public.admin_profiles (
  user_id uuid primary key references auth.users(id) on delete cascade,
  role text not null default 'moderator'
    check (role in ('owner', 'admin', 'moderator')),
  created_at timestamptz not null default now()
);
```

## Helper functions / triggers

### Updated timestamp

```sql
create or replace function public.set_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

create trigger letters_set_updated_at
before update on public.letters
for each row execute function public.set_updated_at();
```

### Like count increment

Option A: handle counts in application server transaction.

Option B: use triggers.

Application transaction is easier to understand during MVP.

Pseudo logic:

```txt
insert into letter_likes(letter_id, visitor_id)
if inserted:
  update letters set hearts_count = hearts_count + 1
```

### Unlike behavior

MVP decision: avoid unlike at first or allow toggle.

Simpler MVP:

- click heart once = liked
- no unlike UI

If toggle is required:

- delete from `letter_likes`
- decrement `hearts_count` safely but never below 0

## Queries

### Public approved letters only

```sql
select *
from public.letters
where status = 'approved'
order by approved_at desc;
```

### Featured

```sql
select *
from public.letters
where status = 'approved'
  and is_featured = true
order by approved_at desc;
```

### Most loved

```sql
select *
from public.letters
where status = 'approved'
order by hearts_count desc, approved_at desc;
```

### Most shared

```sql
select *
from public.letters
where status = 'approved'
order by shares_count desc, approved_at desc;
```

### Latest

```sql
select *
from public.letters
where status = 'approved'
order by approved_at desc;
```

### Random approved letter

Simple MVP:

```sql
select *
from public.letters
where status = 'approved'
order by random()
limit 1;
```

This is fine for small MVP. Later, for large archive, replace with more efficient random selection.

### Find by archive number

```sql
select *
from public.letters
where archive_number = $1
  and status = 'approved';
```

### Find my letters by email

Do not reveal results directly on public page.

Server flow:

1. Normalize email.
2. Query matching rows.
3. Send email with results if any.
4. Show same success message whether matches exist or not.

```sql
select archive_number, status, approved_at
from public.letters
where email_normalized = $1
order by created_at desc;
```

## Row Level Security notes

Recommended MVP approach:

- Public client can read only approved letters.
- Public mutations go through Next.js server routes/actions using server Supabase client.
- Admin mutations require Supabase Auth and admin role check.

If enabling direct public reads via Supabase anon key, RLS policy should allow only approved letters.

Example concept:

```sql
alter table public.letters enable row level security;

create policy "Public can read approved letters"
on public.letters
for select
using (status = 'approved');
```

Admin policies should check `admin_profiles`.

## Privacy

Publicly visible:

- archive number
- letter text
- author display name if not anonymous
- city/country if provided and intended for public display
- language
- approved date
- hearts count
- shares count

Never public:

- email
- admin notes
- rejection reason unless intentionally shown to author by email
- internal UUIDs when avoidable
